# /remember — 沉淀项目记忆

回顾本次会话（$ARGUMENTS 优先），把**下次还会用到的知识**写进 kb。写之前先想：
「三个月后另一个 AI 接手这个项目，这条信息能让它少走弯路吗？」不能就别写。

## 分拣规则（一条信息只进一个去处）

| 类型 | 去处 | 例 |
|---|---|---|
| 新发现的项目事实（带证据） | `ai/kb/PROJECT-FACTS.md` 对应章节 | 「分页其实用的是自制 PageUtil，非 PageHelper —— 见 xxx:12」 |
| 一次性的设计取舍及理由 | `ai/kb/decisions/NNN-<slug>.md`（按模板） | 「为什么 XX 功能沿用旧 BLogic 而没建 Service」 |
| 踩坑教训 / review 高频指摘 | `ai/kb/lessons.md` 追加一行 | 「XX 共通部品会吞异常，调用后必须自查返回码」 |
| 某模块的局部知识 | `ai/kb/modules/<模块>.md` | 「受注模块的状态机、隐藏前提、慎动点」 |
| 规则级结论（今后必须遵守） | `ai/kb/rules/core-contract.md` 相应段 | 「排序列白名单在 XxxConst，新增列要同步」 |

## 保洁义务（每次 remember 顺手做）

- `lessons.md` 超过 ~40 条时：合并同类项；**连续三次 review 都出现的教训升级进 rules**，原条目删除。
- 与既有记录矛盾的新事实：更新原记录（保留证据），不要追加成两条矛盾记录。
- 事实卡的证据 `路径:行号` 若因本次改动失效，顺手修正。

## 收尾

列出本次写入/修改的 kb 文件与条目，供用户过目；kb 变更随代码一起提交。
