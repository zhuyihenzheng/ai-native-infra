# /plan — 生成实装方针

读式样书 `$ARGUMENTS`，产出実装方針書 `ai/work/<ticket>/plan.md`。
**方针要经人 review 后才进 `/implement`**——这对应 SI 流程里「方式設計 → 製造」的关口。

## 前置

- 项目已对齐（`PROJECT-FACTS.md` ≠ PLACEHOLDER）。
- 读式样书 + 它引用的 DEF；**结构（字段/型/桁/列映射/code 值）只来自 DEF，不臆造**；行为来自式样书。
- 没跑过 `/impact` 就先跑（改修案件必须）。
- 读 `core-contract.md`、相关层 examples、涉及模块的 `ai/kb/modules/<模块>.md`（若存在）。

## 方針書结构（写入 `ai/work/<ticket>/plan.md`）

```markdown
# 実装方針 — <ticket> <标题>
> 式样书: <路径> / DEF: <路径> / 日期 / 状态: DRAFT（人未审）

## 1. 変更概要        <!-- 三五行说清做什么、为什么 -->
## 2. 影響範囲        <!-- 引 impact.md 结论；新規开发写「新規」 -->
## 3. 変更方針        <!-- 关键取舍：沿用哪个既存模式(指向example/既存类)、为什么这样切 -->
## 4. 変更ファイル一覧 <!-- 按层列：文件 | 新規/修正 | 变更内容一句话 | 关联式样书条目ID -->
## 5. DB / IF 影響    <!-- 无则写「なし」 -->
## 6. テスト方針      <!-- 新增哪些层的测试、回归范围 -->
## 7. リスク・未確定事項 <!-- 含所有 [assumed]；需要用户裁决的问题列成问句 -->
```

## 纪律

- 方针必须**贴既存模式**：指明「仿照 <既存类>」而不是发明新结构。
- 式样书有歧义 → 写进「未確定事項」问用户，**不要**用自己的解释直接定案。
- 完成后向用户呈现方针要点 + 未確定事項，请其裁决后把状态改为 `APPROVED`。
