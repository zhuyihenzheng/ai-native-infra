# 修改既存代码的安全规则

> 通用规则，**不需对齐即生效，优先级最高**。AI 修改既存企业代码时逐条适用。

## 改之前

1. **影响调查先行**：改共用类 / 表 / 列 / 共通部品 / SQL 片段之前必须 `/impact`。
   影响面超出用户描述的预期时，先报告再动手。
2. **有式样书依据**：改路由、public 签名、SQL 列、表结构、画面项目，必须能指出式样书条目
   或用户明示指示。「我觉得更合理」不是依据。
3. **读懂再改**：改一个方法前读完它的调用方；改 XML SQL 前读完引用它的 Java 侧与 `<include>` 关系。

## 改的时候

4. **最小 diff**：只改任务所需的行。不顺手格式化、不重排 import、不重命名、不删"看起来没用"的代码。
5. **贴合现状**：项目怎么写就怎么写，哪怕它不规范。发现坏味道 → 记入 `lessons.md` 或报告用户，
   不在业务改动里夹带重构。重构必须是独立的、用户明示批准的任务。
6. **不用脚本批量重写**：禁止 sed/正则批量改未逐个确认过的代码位置。
7. **兼容优先**：既存公开接口（被其他模块/系统调用的）默认只加不改；确需破坏性变更时列出所有调用方并请用户确认。
8. **可疑即停**：遇到与事实卡矛盾的代码、标 `[assumed]` 的规则成为改动依据、或需要删除数据/改权限/碰凭证——停下问用户。

## 改完之后

9. **越界自查**：`git diff --stat` 对照计划的文件清单，多出来的文件要么有解释要么撤销。
10. **verify 收口**：跑 `bash ai/tools/verify.sh`，红/绿如实报告。「编译通过」不等于验证；
    涉及画面/接口的改动按测试方针补行为验证。
11. **可追溯**：新代码挂 traceability ID 回链式样书；改了式样书/DEF 就全仓搜该 ID，连带更新 testcases/测试/fixture。
